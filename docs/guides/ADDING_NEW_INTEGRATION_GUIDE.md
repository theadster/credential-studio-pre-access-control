# Adding New Integration Guide

## Overview

This guide provides a comprehensive, step-by-step procedure for adding new integrations to credential.studio. Follow each phase in order to ensure proper implementation, security, and maintainability.

**Estimated Time:** 2-4 hours for a complete integration

**Prerequisites:**
- Understanding of the [Integration Architecture Guide](./INTEGRATION_ARCHITECTURE_GUIDE.md)
- Familiarity with [Integration Patterns Reference](./INTEGRATION_PATTERNS_REFERENCE.md)
- Access to Appwrite console
- Node.js and npm installed locally

**What You'll Build:**
- Database collection for integration configuration
- Backend functions for CRUD operations
- API integration for event settings
- Frontend UI components
- Environment variable configuration
- Tests and verification

## Phase 1: Database Setup

**Goal:** Create the Appwrite collection to store integration configuration data.

### 1.1 Define Your Integration Schema

First, determine what configuration fields your integration needs.

**Example Schema for "ExampleService" Integration:**
```typescript
interface ExampleServiceIntegration {
  $id: string;              // Auto-generated by Appwrite
  eventSettingsId: string;  // Required - links to event_settings
  version: number;          // Required - for optimistic locking
  enabled: boolean;         // Required - enable/disable toggle
  
  // Integration-specific fields
  apiEndpoint: string;      // Example: API endpoint URL
  webhookUrl: string;       // Example: Webhook callback URL
  templateId: string;       // Example: Template identifier
  autoSync: boolean;        // Example: Auto-sync feature flag
  
  // Appwrite metadata (auto-generated)
  $createdAt: string;
  $updatedAt: string;
}
```

**⚠️ Security Note:** Never store API keys, secrets, or credentials in the database. Use environment variables instead.

### 1.2 Create Collection in Appwrite Console

**Manual Steps:**

1. Log into Appwrite Console
2. Navigate to your database
3. Click "Add Collection"
4. Set Collection ID: `example_service` (use snake_case)
5. Set Collection Name: "Example Service Integration"

**Or use the setup script (recommended):**

Edit `scripts/setup-appwrite.ts` and add your collection:

```typescript
// Add to COLLECTIONS constant

}
```

### 1.4 Create Indexes

Create an index on `eventSettingsId` for efficient queries.

```typescript
// In setup-appwrite.ts, add to createIndexes function

  // ... existing indexes
  
  // Example Service Integration Index

    databaseId,
    COLLECTIONS.EXAMPLE_SERVICE,
    'eventSettingsId_idx',
    'key',
    ['eventSettingsId'],
    ['ASC']
  );

```

**Why This Index?** Every integration query filters by `eventSettingsId`, so this index is critical for performance.

### 1.5 Set Permissions

Configure collection permissions for security.

**Recommended Permissions:**
- **Create:** Authenticated users (will be validated in API)
- **Read:** Authenticated users
- **Update:** Authenticated users (will be validated in API)
- **Delete:** Authenticated users (will be validated in API)

**Manual Steps:**
1. In Appwrite Console, go to your collection
2. Click "Settings" → "Permissions"
3. Add permission: `role:all` with Read, Create, Update, Delete
4. Save changes

**⚠️ Security Note:** API routes will enforce role-based permissions. Collection permissions allow authenticated access, but API validates specific user roles.

**Environment Variable Pattern:** Use `NEXT_PUBLIC_APPWRITE_[NAME]_COLLECTION_ID` format.

### 2.3 Implement Getter Function

Create a function to fetch the integration:

```typescript
/**
 * Get Example Service integration for an event
 * @param databases Appwrite Databases instance
 * @param eventSettingsId Event settings ID to query
 * @returns ExampleServiceIntegration or null if not found
 */
export async function getExampleServiceIntegration(
  databases: Databases,
  eventSettingsId: string
): Promise<ExampleServiceIntegration | null> {
  try {
    const response = await databases.listDocuments(
      DATABASE_ID,
      EXAMPLE_SERVICE_COLLECTION_ID,
      [Query.equal('eventSettingsId', eventSettingsId), Query.limit(1)]
    );
    
    return response.documents.length > 0 
      ? (response.documents[0] as any) 
      : null;
  } catch (error: any) {
    // Handle collection not found gracefully
    if (
      error.code === 404 || 
      error.code === 'document_not_found' || 
      error.code === 'collection_not_found'
    ) {
      return null;
    }
    
    console.error('Error fetching Example Service integration:', error);
    throw error;
  }
}
```

**Getter Pattern Notes:**
- Always query by `eventSettingsId`
- Use `Query.limit(1)` for efficiency
- Return `null` for 404 errors (collection/document not found)
- Throw other errors for proper error handling
- Add JSDoc comments for documentation

### 2.4 Implement Update Function with Optimistic Locking

Create a function to update the integration:


```typescript
/**
 * Update Example Service integration with optimistic locking
 * @param databases Appwrite Databases instance
 * @param eventSettingsId Event settings ID
 * @param data Partial integration data to update
 * @param expectedVersion Expected version for optimistic locking (optional)
 * @returns Updated ExampleServiceIntegration
 * @throws IntegrationConflictError if version mismatch
 */
export async function updateExampleServiceIntegration(
  databases: Databases,
  eventSettingsId: string,
  data: Partial<Omit<ExampleServiceIntegration, '$id' | 'eventSettingsId' | 'version'>>,
  expectedVersion?: number
): Promise<ExampleServiceIntegration> {
  return updateIntegrationWithLocking<ExampleServiceIntegration>(
    databases,
    EXAMPLE_SERVICE_COLLECTION_ID,
    'ExampleService',
    eventSettingsId,
    data,
    expectedVersion,
    () => getExampleServiceIntegration(databases, eventSettingsId)
  );
}
```

**Update Pattern Notes:**
- Use `updateIntegrationWithLocking` helper for consistency
- Omit `$id`, `eventSettingsId`, and `version` from update data type
- Pass getter function as last parameter
- Optimistic locking is handled automatically

### 3.2 Add Integration Update Call

In the PUT handler, add your integration update:

```typescript
// In PUT handler, after core event settings update
try {
  // Update integrations in parallel
  const integrationUpdates = [];

  // Existing integrations...

  // Example Service Integration
  if (integrationFields.exampleService) {
    integrationUpdates.push(
      updateExampleServiceIntegration(
        databases,
        eventSettingsId,
        integrationFields.exampleService
      ).catch((error) => {
        console.error('Failed to update Example Service integration:', error);
        return null;
      })
    );
  }

  // Wait for all integration updates
  await Promise.all(integrationUpdates);

  // Fetch updated settings with integrations
  const updatedSettings = await flattenEventSettings(databases, updatedEventSettings);

  return res.status(200).json(updatedSettings);
} catch (error: any) {
  console.error('Error updating event settings:', error);
  return res.status(500).json({ 
    error: 'Failed to update event settings',
    details: error.message 
  });
}
```

**Update Pattern:**
- Add to parallel update array
- Catch and log errors (don't fail entire request)
- Return flattened settings with all integrations

### 3.3 Add Validation (Optional)

Add validation for integration-specific fields:

```typescript
// Before updating, validate Example Service fields
if (integrationFields.exampleService?.enabled) {
  const { apiEndpoint, webhookUrl } = integrationFields.exampleService;

      error: 'Invalid Example Service API endpoint URL'

    return res.status(400).json({
      error: 'Invalid Example Service webhook URL'
    });

// Helper function
function isValidUrl(url: string): boolean {
  try {
    new URL(url);
    return true;
  } catch {
    return false;
  }
}
```

**Validation Guidelines:**
- Validate before updating database
- Return 400 status for validation errors
- Provide clear error messages
- Only validate when integration is enabled

### 5.3 Add to Environment Validation

If you have environment validation (recommended), add your variables:

Create or edit `src/lib/env.ts`:

```typescript
const requiredEnvVars = [
  'NEXT_PUBLIC_APPWRITE_ENDPOINT',
  'NEXT_PUBLIC_APPWRITE_PROJECT_ID',
  // ... other required vars
];

const optionalEnvVars = [
  'EXAMPLE_SERVICE_API_KEY',
  'EXAMPLE_SERVICE_API_SECRET',
  'NEXT_PUBLIC_APPWRITE_EXAMPLE_SERVICE_COLLECTION_ID',
];

export function validateEnv() {
  const missing = requiredEnvVars.filter(
    (varName) => !process.env[varName]
  );

  if (missing.length > 0) {
    throw new Error(
      `Missing required environment variables: ${missing.join(', ')}`
    );
  }

  // Warn about missing optional vars
  const missingOptional = optionalEnvVars.filter(
    (varName) => !process.env[varName]
  );

  if (missingOptional.length > 0) {
    console.warn(
      `Missing optional environment variables: ${missingOptional.join(', ')}`
    );
  }
}
```

## Common Pitfalls & Solutions

### Pitfall 1: Forgetting to Add to flattenEventSettings

**Problem:** Integration data not available in frontend

**Solution:** Always add your integration to `flattenEventSettings` function

**How to Detect:** Frontend shows undefined values for integration fields

### Pitfall 2: Not Handling Null Integrations

**Problem:** Errors when integration doesn't exist

**Solution:** Use `??` operator for default values everywhere

**Example:**
```typescript
// ❌ Wrong
const endpoint = formData.exampleServiceApiEndpoint;

// ✅ Correct
const endpoint = formData.exampleServiceApiEndpoint ?? '';
```

### Pitfall 3: Storing Credentials in Database

**Problem:** Security vulnerability

**Solution:** Always use environment variables for credentials

**How to Detect:** Check database documents in Appwrite Console

### Pitfall 4: Missing Index on eventSettingsId

**Problem:** Slow queries

**Solution:** Always create index on eventSettingsId

**How to Detect:** Check query execution time in Appwrite Console

### Pitfall 5: Not Using Optimistic Locking

**Problem:** Concurrent updates overwrite each other

**Solution:** Use `updateIntegrationWithLocking` helper

**How to Detect:** Test concurrent updates in multiple browser tabs

### Pitfall 6: Inconsistent Field Naming

**Problem:** Confusion and bugs

**Solution:** Use consistent naming pattern:
- Database: snake_case
- TypeScript interface: camelCase
- Frontend (flattened): integrationNameFieldName

### Pitfall 7: Not Disabling Fields When Integration Disabled

**Problem:** Poor UX

**Solution:** Always disable form fields when `enabled` is false

**Example:**
```typescript
<Input
  disabled={!formData.exampleServiceEnabled}
  // ... other props
/>
```

### Pitfall 8: Missing Validation

**Problem:** Invalid data in database

**Solution:** Validate in API before saving

**Example:**
```typescript
if (data.apiEndpoint && !isValidUrl(data.apiEndpoint)) {
  return res.status(400).json({ error: 'Invalid URL' });
}
```

### Pitfall 9: Not Testing Error Cases

**Problem:** Crashes in production

**Solution:** Test all error scenarios:
- Missing environment variables
- Invalid data
- Network errors
- Concurrent updates
- Missing integrations

### Pitfall 10: Forgetting to Update Documentation

**Problem:** Future developers confused

**Solution:** Always update:
- `.env.example`
- Integration-specific guide
- This guide (if pattern changes)
- API documentation
